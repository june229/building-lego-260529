## 1. 커밋 메시지 규칙 (Commit Convention)

커밋 메시지는 널리 쓰이는 Angular 커밋 규칙을 기반으로 작성하며, 어떤 작업을 수행했인지 직관적으로 파악할 수 있어야 합니다.

> 📌 **구조:** `타입: 한 줄 요약 (#이슈번호)`

### 🛠️ 커밋 타입(Type) 목록
* `feat`: 새로운 기능, 문서, 이미지 등을 프로젝트에 추가할 때
* `fix`: 잘못된 내용 수정, 누락된 부품 목록 수정 등 오류를 수정할 때
* `docs`: 코드나 문서의 단순 오타 수정, 내용 보완 등 (README.md, 매뉴얼 등)
* `chore`: 폴더 구조 변경, 파일 위치 이동, 프로젝트 설정 변경 등

### 📝 작성 예시
* `feat: 레고 부품 체크리스트 문서 추가 (#1)`
* `docs: README 파일에 레고 완공 사진 추가 (#4)`
* `fix: 조립 설명서 오타 및 누락 번호 수정 (#3)`

---

## 2. 브랜치명 규칙 (Branch Naming Convention)

브랜치명은 어떤 이슈에 대한 작업인지 식별하기 위해 **작업 타입**, **이슈 번호**, **요약 내용**을 조합하여 생성합니다.

> 📌 **구조:** `작업타입/#이슈번호-간단한작업명`

### 🛠️ 작업 타입(Type)
* `feature/`: 새로운 작업 또는 기능을 개발할 때
* `fix/`: 오류나 잘못된 사항을 긴급하게 수정할 때

### 📝 작성 예시
* `feature/#1-purchase-list`
* `feature/#2-setup-environment`
* `feature/#3-assembly-main`
* `fix/#3-missing-parts`

---

## 3. PR 및 코드 리뷰 규칙 (Pull Request & Code Review)

모든 작업은 `main`(또는 `develop`) 브랜치에 직접 커밋할 수 없으며, 반드시 PR을 거쳐 동료들의 검토를 받아야 합니다.
